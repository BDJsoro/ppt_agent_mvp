# 개발 워크플로우 및 자동 에러 복구(Feedback Loop) 지침

이 문서는 AI 웹 에이전트(주로 Gemini 계열 플래시 모델)가 이 프로젝트의 코드를 작성하고 개발해 나갈 때 반드시 따라야 하는 행동 강령과 규칙입니다. AI 개발자는 작업을 시작할 때 항상 이 문서들을 먼저 읽고 숙지해야 합니다.

## 1. 개발 마일스톤 (단계별 구현 스텝)

- **Phase 1: 인프라 및 환경 세팅**
  - Supabase Auth 및 테이블(유저, 템플릿, 히스토리) 구축
  - Next.js 및 FastAPI 보일러플레이트 세팅

- **Phase 2: 코어 엔진 개발 (AI + 파워포인트)**
  - Gemini API 구조화 프롬프트 작성 (슬라이드 수 조절 기능 포함)
  - `python-pptx`를 이용한 텍스트 매핑, 폰트 적용, 숨김 메타데이터 지정 단일 테스트 통과

- **Phase 3: 비동기 큐잉 및 대시보드 통합**
  - Celery 기반 대기열 작업 구현 및 로딩 UI(Progress) 연결
  - 히스토리 데이터를 불러와 마이페이지 대시보드 표 렌더링 구현

## 2. 🚨 에러 대응 루프 (Error Feedback & Retry Loop) 기본 스크립트

AI 에이전트가 코드를 작성한 뒤 테스트를 실행하고 터미널이나 과정 속에서 `Error/Exception` 로그를 만났을 경우, 멈추지 말고 다음의 **디버깅 자동화 루프(1~4단계)**를 반드시 가동하십시오.

1. **상황 인식 (Capture):**
   - 터미널이나 서버 응답에서 뱉어낸 에러 스택(Error StackTrace)을 확인한다.
2. **작업 및 에러 로그 기록 (Context Memory):**
   - 에러가 연달아 발생할 경우 AI가 같은 실수를 반복("똑같은 코드로 무한 수정")하는 것을 막기 위해, 루트 디렉토리의 임시 파일(예: `.ai_error_log.md`)에 '현재 발생한 에러'와 '내가 시도했던 실패한 패치 방식'을 누적해서 기록한다.
3. **가설 수립 (Hypothesis Formulation):**
   - 기록된 로그 분석을 바탕으로 "이전에 B 방식으로 수정했을 때 안 되었으니, 이번에는 C 객체 구조를 뜯어고쳐야겠다"처럼 겹치지 않는 새 가설을 수립한다.
4. **코드 자체 패치 (Patch & Re-run):**
   - 에러의 원인부를 수정하고, 로그 디버깅 코드(`print` 혹은 `console.log`)를 달아 테스트를 재실행한다.
5. **성공 여부 판단 (Retry Condition):**
   - **Case A (성공):** 해결 시, 다음 마일스톤으로 넘어간다.
   - **Case B (에러 재발생):** `.ai_error_log.md`에 현 시도의 실패 내역을 적고, 최대 3~4회까지만 이 루프(로그기록-새 가설-패치-확인)를 반복한다.
   - **Case C (한계 봉착):** 3~4회 반복 후에도 해결 불가 시 작업을 즉시 멈추고, 작성 누적된 `.ai_error_log.md` 파일 정보를 활용해 사용자(User)에게 상황 브리핑 후 개입을 요청한다.

## 3. GitHub 버전 관리 (Version Control) 규칙

규모가 큰 프로젝트이므로, 안전한 롤백(Rollback)과 히스토리 관리를 위해 AI 에이전트는 다음 규칙에 따라 깃허브 커밋/푸시를 주도적으로 제안해야 합니다.

1. **자동 커밋 금지 (Auto-commit X):** 코드 수정 시마다 무조건 커밋하면 히스토리가 오염됩니다. 임의로 백그라운드에서 자동 커밋하지 마십시오.
2. **기능(마일스톤) 단위 완료 시 승인 요청 (권장):** 특정 기능 단위(예: "API 연동 성공", "UI 렌더링 완료", "에러 버그 픽스 성공")가 정상 작동한다고 확인된 직후, AI는 즉시 다음 작업을 넘어가기 전에 사용자에게 묻습니다.
   - *"해당 기능이 성공적으로 구현 및 테스트되었습니다. 안전을 위해 현재까지의 작업물을 GitHub에 커밋하고 푸시(Push)할까요?"*
3. **명확한 커밋 메시지 작성:** 사용자가 "승인(Yes)"하면, AI는 `git add .` 커밋 내역을 분석하여 의미 있는 메시지(`git commit -m "feat: PPT 메타데이터 삽입 로직 추가"`)를 작성해 깃허브 명령어(`run_command`)를 실행합니다.

## 4. 자체 품질 검증 및 PM(사용자) 보고 의무 (Auto-QA)

PM(사용자)이 비개발자이므로, 모든 기술적 검증과 예외 처리의 책임은 전적으로 AI 에이전트에게 있습니다. AI 에이전트는 단위 기능(Task) 코딩을 완료했다고 판단할 때마다 **절대 바로 넘어가지 말고, 반드시 아래 4가지 항목을 스스로 자가 진단한 리포트를 작성하여 사용자에게 먼저 승인을 요청**해야 합니다.

*   [보고 필수 양식] AI는 완료 보고 시 다음 4가지 관점 리포트를 렌더링합니다:
    1.  **대안 기술 비교:** "현재 코드에 사용한 X방식 외에 Y방식이 있지만, 성능/유지보수 측면에서 X를 택했습니다."
    2.  **엣지 케이스 (Unhappy Path) 방어:** "사용자 오입력이나 네트워크 단절 시 발생할 최악의 시나리오와, 이를 막기 위해 제가 이번 코드에 넣어둔 예외 처리(방어막) 내역입니다."
    3.  **결과물 퀄리티 보장:** "단순 동작을 넘어, 응답 퀄리티(또는 UI 정확도, 리서치 품질)를 100% 보장하기 위해 제가 추가로 설정해 둔 코드 제약 조건입니다."
    4.  **테스트 증명:** "이 코드가 완벽히 구동됨을 증명하기 위해 제가 방금 실행해 본 터미널 테스트 명령어와 그 성공 로그(출력값)입니다."

**※ AI 에이전트 최종 확인 사항:**
임의로 주요 비즈니스 스택(FastAPI, Next.js, 8장 슬라이드 제한 기획, 파일 메타데이터 삽입 등)을 회피하지 마십시오. 설정된 기획 문서를 철저히 맹신하고 이를 완수하는 방향으로 코드를 전개해야 합니다.
